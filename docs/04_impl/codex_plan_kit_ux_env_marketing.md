# 実装計画 v2 — Create kit UX改善・環境固定・マーケティング（Codex 引き継ぎ）

## 概要

本計画は **Codex（コーディングエージェント）が実装・テストを担当する** 前提の引き継ぎドキュメント。
2026-06-12〜13 のアップリフト（LP刷新・モーションシステム・サンプルデッキv2・デモエージェント・
ライブポーリング・品質ゲート・Cmd+Kパレット・初回ツアー）の **後続フェーズ** を定義する。

製品哲学は不変: **依存ゼロ・ローカルファースト・生成は外部サービスに委ねる（本体は絶対に生成APIを呼ばない）**。

### 外部サービス・リファレンスの適用可否

`ref-google-cloud` / `ref-supabase` / `ref-vercel-deploy` / `ref-gemini-api` は **本プロジェクトに適用しない**
（依存ゼロのローカル Node ツールであり、DB・OAuth・サーバーレスデプロイが存在しないため。
詳細は `implementation_plan.md` の同名セクション参照）。外部接点は npm / GitHub（Pages・Release）と、
Phase 4 のマーケティング投稿先（X / note / dev.to 等 — すべて 👤 が投稿）のみ。

### 担当凡例

| 記号 | 担当者 | 説明 |
|------|--------|------|
| 👤 | 人間 | 投稿・アカウント操作・最終視覚確認・録画 |
| 🤖 | Codex | コーディング・テスト・ドキュメント生成 |
| 🤖S | Codex のサブエージェント | **新規コンテキストで起動する検証専用エージェント**（実装者と分離） |
| 👤+🤖 | 共同 | 🤖が実行し👤が確認・承認 |

### 品質担保の大原則（全フェーズ共通・必読）

1. **実装と検証の分離**: 各タスク完了時、実装した Codex 自身の確認に加えて、
   **コンテキストを持たないサブエージェント（🤖S）に検証を依頼する**。サブエージェントには
   「やったこと」を教えず、**受け入れ条件のみ** を渡して合否判定させる（実装者バイアスの排除）。
2. **毎タスク必須コマンド**（1つでも落ちたらタスク未完了）:
   ```bash
   npm run check                       # 構文チェック（server/app/gallery/scripts）
   npm test                            # 2026-06-13 時点 38 件全パス + 自分が追加した分
   node server.mjs --workspace ./workspace/demo --init sample --doctor   # 公開健全性
   ```
3. **i18n パリティ**: UI 文言を追加したら必ず `t()` 経由とし、EN/JA **両方の I18N マップ** に
   キーを追加。検証: 両マップのキー数・キー集合の一致を node ワンライナーで確認する
   （app.js から I18N の ja/en オブジェクトを抽出して `Object.keys` を比較。差分ゼロが合格）。
4. **CSP/オフライン**: 外部リソース読み込み・インラインscript属性の追加は禁止
   （`default-src 'self'`）。検証: 起動した各ページの HTML に `http(s)://` のリソース参照が
   ないことを grep。
5. **契約の不変**: `image-arranger-request.v1`（docs/request-spec.md）と deck スキーマは
   **後方互換を壊さない**。フィールド追加は可、変更・削除は不可。
6. **既存機能のデグレ禁止**: renderT()（View Transitions）・livePoll（5秒ポーリング）・
   Undoトースト・品質ゲート・Cmd+Kパレット・初回ツアーを壊さないこと。タッチした画面は
   reduced-motion / キーボード操作 / Escape を必ず再確認。
7. **作業ログ**: 各フェーズの実装ログ・スクリーンショットを `agent-logs/<日付>-codex-<phase>/`
   に保存（リポジトリ規約。`agent-logs/` は gitignore 済み）。
9. **UIに内部メッセージを出さない（2026-06-13 追加・恒久原則）**: ユーザー向けUI（画面文言・
   トースト・パネル・プレースホルダ）に、内部実装の説明を**出さない・今後も勝手に追加しない**。
   禁止例: コマンド名（`npm run demo-agent` 等）、「エージェント」「依頼JSON」「request file」
   などの内部用語、処理アーキテクチャの説明、開発者向け注記。ユーザーに見せてよいのは
   「何ができるか・いま何が起きているか・次に何をすればよいか」を生活者の言葉で書いたものだけ。
   内部向け情報の置き場は README / AGENTS.md / docs（開発者文書）に限る。
8. **テストデータの安全（2026-06-13 追加・違反厳禁）**: ユーザーの正データは
   `workspace/demo-aichan/` のみで、これとworkspace/archive/ には一切触れない（読取・変更・
   テスト利用・サーバー起動すべて禁止）。検証は必ず `/tmp/ia-test-*` ワークスペース＋
   **ポート4901-4999** で行い、終了時にサーバー停止・/tmp削除・**開いたブラウザタブを全て閉じる**。
   Aoiサンプルは架空のデモデータであり、ユーザーのデータとして提示しない。
   詳細は `workspace/_LOCAL_RULES.md`（gitignored）と AGENTS.md の
   「Workspace & test-data safety」を必読。

## Phase 1: Create kit UX改善 ✅

**目標**: Create kit タブの「A. シートを作る」「B. パーツに分解」の操作の分かりにくさを解消する。

**現状の実装箇所（実測）**:
- レンダリング: `public/app.js` の kit タブ描画（`renderKit` 相当、〜1820行付近）
- ルートA: シート名 + テンプレ選択（`BUILTIN_SHEET_TEMPLATE`、app.js:1760）+ プロンプト textarea + 「キューに入れる」
- ルートB: キャラ名 + 追加要望 + 「解析を依頼」→ 完了後 `state.kitResults` →「パーツを選ぶ」→ チェックボックス → 取り込み
- パーツ語彙: `prompts.mjs` の `BASE_KIT_PARTS`（**`palette` キーが既に存在** — Phase 2 で使う）

**分かりにくさの仮説（コードレビューより。Phase 1 開始時に 🤖S の初見テストで検証してから着手すること）**:
1. ルートA/B が横並びで同時表示され、**どちらを・いつ使うべきか** の説明がない
2. ボタンを押した後 **何が起きたのか見えない**（リクエストは Queue タブに行くが、その場では小さな行が出るだけ。処理は外部エージェント任せという製品の核心が初見では理解不能）
3. ルートB の結果フロー（解析依頼 → 完了 → kitResults 一覧 →「パーツを選ぶ」→ プレビュー → 取り込み）が **画面のあちこちに散在**
4. ソース選択（手順1）は「採用済み画像」が前提だが、無い場合の空状態から **次の行動への導線が弱い**
5. ルートA のプロンプトが巨大な textarea で初見に威圧的

## Phase 2: カラーパレット統合 ✅

**目標**: 「シートを作る」（ルートA）にカラーパレットを組み込み、生成されるシートの色再現品質を上げる。

**設計方針（実装前に必ず現状確認）**:
- `BASE_KIT_PARTS` には既に `palette` パーツ（カラースウォッチグリッド）が存在し、ルートBで
  パレット参照画像を作れる。**「パレット参照を持っているならシート生成時に自動同梱し、
  プロンプトで色の正(authority)として扱わせる」** のが本フェーズの核心。
- 生成サービス非依存を維持: パレットは **参照画像＋プロンプト文言** としてリクエストに載せるだけ。
  本体で色抽出などの画像処理はしない（依存ゼロ維持。`placeholder-art.mjs` 流用の軽量実装なら可）。

## Codex からの確認事項への回答（2026-06-13）

### Q1. 🤖S 検証は実際に別サブエージェントを起動するのか、「新規コンテキスト相当の検証ログ」でよいか

**仕組みは問わない。守るべき性質は「コンテキスト分離」**。優先順位:

1. サブエージェント機構が使えるなら実際に起動する（最優先）
2. 使えない場合は **別の新規 Codex セッション/実行** で代替してよい。ただし検証者に渡してよいのは
   (a) 受け入れ条件、(b) 本計画書記載のプロトコル文、(c) リポジトリへのアクセス、の3点のみ。
   実装時の会話・diff の説明・「こう直した」という情報を渡してはならない
3. **実装した本人（同一コンテキスト）が書いた「新規コンテキスト相当のログ」は不合格**。
   これは検証の意味（実装者バイアスの排除）を失うため

検証ログの先頭に「検証者に与えた入力一式」を必ず記録すること。これで分離が監査可能になる。

### Q2. palette 検出のための明示フィールド（`partKey` / `tags`）を追加してよいか

**追加してよい。`partKey` フィールドを推奨**（追加のみ・後方互換維持なので契約ルール違反にならない）:

- ルートB のパーツ取り込み時、**全パーツ**（palette に限らず）にエントリへ
  `partKey: "<BASE_KIT_PARTS の key>"` を永続化する。deck スキーマは未知フィールドを
  保持する設計（`normalizeState` で消えないことをテストで確認すること）
- `tags` での明示化は採用しない。tags はユーザーが編集・削除できる表示用データであり、
  機能判定の根拠には不向き（`"base-kit"` タグの既存用法は表示チップ用なので現状維持）
- **検出順序**: ① `entry.partKey === "palette"` → ② レガシー救済として id 接頭辞
  `base-kit-palette` 一致（既存デッキ向けフォールバック）。この2段で実装する
- request-spec.md の更新は不要（deck 内部のフィールドであり、リクエスト契約には載らない）

### Q3. 「先にパレットを作る」はルートB経由か、palette 専用の直接生成か

**palette 専用の直接生成リクエストを1クリック投入する（後者）を正とする**。理由:

- CTA の目的は「パレットを持っていない人を最短でパレット保有状態にする」こと。
  ルートB は分解元のシート画像が前提で、ステップも多く、目的に対して遠回り
- 実装: `BASE_KIT_PARTS` の palette エントリの hint からプロンプトを自動組み立てし、
  そのキャラの**採用済み参照**（master / 顔があれば優先）を `inputs.refImages` に添付して
  `generate` リクエストを投入（色の正は既存の採用済みアートから導く）
- **採用済み参照が1枚も無いキャラでは直接生成を出さない**。色の根拠が無いパレットは
  品質向上に寄与しないため、その場合は「先に Base タブで候補を採用する」導線
  （Phase 1 タスク 1.5 の空状態CTA）へ誘導する
- ルートB（シートから分解して palette を得る）は従来どおり並存。CTA の補足文で
  「手元にシート画像があるなら B ルートでも作れる」と1行案内する

### Q4. `round2-research.md` が見えない。デモ動画絵コンテはどうするか

**リポジトリ内に昇格済み: `docs/marketing/video-shotlist.md` を正とする**（本回答と同時にコミット）。

- 元ファイルは gitignore 配下の作業ログだったため、Phase 4 で必要な45秒・9ショットの
  絵コンテ部分を tracked ファイルとして移植した。当時指摘されていた撮影ブロッカー
  （G1 ライブ更新なし 等5件）は **すべて解消済み** である旨も追記してある
- README / LP / 既存 assets からの再構成・調整は **歓迎**。ただし shot 構成を変える場合は
  video-shotlist.md を更新してから Phase 4 のドラフトに反映すること（ドキュメントが正）
- 本計画書の frontmatter と Phase 4 素材表の参照先も video-shotlist.md に更新済み

### 6R.2 レイアウト再設計（改訂2 — オーナーFB「情報量が多すぎる」反映）

**情報設計の原則（最重要）**: 各セクションの既定表示は「**何ができるか1行 ＋ 操作1つ**」だけ。
設定・プロンプト・専門概念はすべて折りたたみの中に隠し、必要な人だけが開く。

```
┌ 素材作成 ─────────────────────────────────────┐
│ 1. コア画像（マスター）                            │

│                                                  │

│   選んだ画像からキャラクター設定シートを作り、        │

└──────────────────────────────────────────┘
```

確定事項（オーナーFB 2026-06-13 反映）:

- **A の既定表示は「キャラクター設定シートを作ってマスターに登録する」だけ**。

  生成結果を個別パーツのベース参照と照合し、差分があれば再生成を依頼する — という

  作り直しを依頼。最大N回）」の平易な語に変更する
